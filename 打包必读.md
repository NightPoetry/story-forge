# 打包必读

给新会话的 AI：你要帮用户完成一次发布打包。按下面的步骤依次执行，不要跳步。

---

## 第一步：确认代码状态

```bash
git status
git log --oneline -5
```

- 如果有未提交的改动，先问用户要不要一起提交。
- 如果有未推送的 commit，先 `git push origin main`。

## 第二步：更新版本号

1. 读 `src-tauri/tauri.conf.json`，找到当前 `"version"` 字段。
2. 读最近的 git tag：`git tag --sort=-v:refname | head -5`。
3. 确定新版本号（通常 patch +1，有新功能则 minor +1），**tauri.conf.json 的 version 必须和即将打的 tag 名一致**。
4. 修改 `src-tauri/tauri.conf.json` 里的 version。
5. 提交：
```bash
git add src-tauri/tauri.conf.json
git commit -m "chore: bump version to vX.Y.Z"
git push origin main
```

## 第三步：打 tag 触发 CI

```bash
git tag vX.Y.Z
git push origin vX.Y.Z
```

tag 推送后，GitHub Actions 会自动构建三个平台（macOS / Linux / Windows）。

## 第四步：确认构建

告诉用户：
- 构建进度：https://github.com/NightPoetry/story-forge/actions
- 产物下载：https://github.com/NightPoetry/story-forge/releases（构建完成后为草稿状态，需手动确认发布）

---

## 如果 push 失败（SSH 连接问题）

重试一次即可，偶尔网络抖动。如果持续失败：
```bash
# 切换到 443 端口
ssh-keyscan -p 443 ssh.github.com >> ~/.ssh/known_hosts
```
并确认 `~/.ssh/config` 里有：
```
Host github.com
  Hostname ssh.github.com
  Port 443
  User git
```

---

## 关键约束（不要踩的坑）

- **version 和 tag 必须一致**：`tauri.conf.json` 的 `version` 字段要和 git tag 名（去掉 v 前缀）完全一致。
- **不要改 workflow 文件**，除非用户明确要求。当前配置已验证可用。
- **Windows 用 NSIS 不用 MSI**：WiX 对中文 productName 会崩，workflow 里已配置 `--bundles nsis`。
- **不要改 tauriScript**：`tauri-action` 会自动追加 `build` 子命令。

---

## 本地 Mac 打包（用户要求时才做）

```bash
npm run tauri:build
# 产物在 src-tauri/target/release/bundle/
```
