## 已完成的改动

### 1. 伏笔批量生成去重（v2.2.1 核心修复）

**问题**：用户通过对话让 AI 批量生成伏笔时，AI 不会检查已有伏笔，导致：
- 生成与已有伏笔语义相同或近似的条目
- 每次生成都是纯累加，越用越混乱
- AI 不知道 `add_foreshadowings` 是"追加"而非"替换"

**修复方案**：三层防御，从 prompt 到代码逐层拦截。

#### 第一层：工具描述强化（`src/api.ts`）

| 改动位置 | 修改内容 |
|----------|---------|
| `ADD_FORESHADOWING_TOOL.description` | 从"批量添加"改为"仅添加新伏笔，不会替换或修改已有伏笔"，要求调用前检查伏笔档案 |
| `items` 字段 description | 强调"仅填写全新的伏笔，不要包含伏笔档案中已有的内容" |
| `secret` 字段 description | 增加"不得与伏笔档案中已有伏笔的隐藏真相重复或近似" |
| `TOOL_GUIDANCE` 伏笔指引 | 增加"先检查伏笔档案，只添加与已有伏笔不重复的全新伏笔" |

#### 第二层：上下文注入（`src/api.ts` — `buildDynamicContext`）

在伏笔档案标题下新增醒目提示：

```
> 以下伏笔已存档，add_foreshadowings 只能追加全新伏笔，不得重复或覆盖已有条目。
```

AI 每次收到上下文时都会看到这行提示，紧贴着已有伏笔列表。

#### 第三层：代码级去重（`src/components/ChatPanel.tsx`）

新增 `isDuplicateForeshadowing()` 函数，作为 prompt 层的安全网：

```typescript
function isDuplicateForeshadowing(newSecret: string, existing: { secret: string }[]): boolean
```

- 标准化处理：去除空白、转小写后比较
- 完全匹配：标准化后相同 → 重复
- 包含匹配：两个 secret 长度均 >10 字符且一方包含另一方 → 重复
- 在主生成（`onAction` 回调）和自动初始化（`runAutoInit` 回调）两个入口都应用过滤

## 关键设计决策与原因

### 为什么用三层防御而不只改 prompt？

Prompt 是建议而非约束——模型可能忽略。代码级去重是硬性保障：即使 AI 返回了重复伏笔，`isDuplicateForeshadowing` 也会静默过滤掉。三层各有作用：
- Prompt 层：减少 AI 产生重复的概率（最佳情况下代码层不需要工作）
- Context 层：让 AI 在看到已有伏笔列表的同时看到"不许重复"的提醒
- Code 层：兜底，任何情况下都不会写入重复数据

### 去重阈值为什么用包含匹配而不用编辑距离？

编辑距离对中文效果差（一个字的差异可能改变整个语义），且计算成本高。包含匹配针对最常见的重复模式：AI 用不同措辞复述同一个 secret，但核心表述一定互相包含。10 字符的最低长度门槛避免了短词误判。
